# План Реализации MVP

Цель плана — реализовывать систему постепенно, чтобы после каждого этапа оставался рабочий срез, который можно запустить локально и проверить.

## Принципы Выполнения

- Каждый этап выполняется в отдельной ветке.
- Для каждого этапа используется `planning/stage-checklist.md`.
- Перед началом этапа выполняется preflight по техническому долгу, инструментам и фиче-реквестам.
- Перед реализацией этапа нужен краткий план с декомпозицией и подтверждение.
- Нужно разделять "срез готов к проверке" и "этап закрыт".
- Срез готов к проверке после реализации, проверок, актуализации документации и коммита в ветке.
- Merge в `master` выполняется после подтверждения, что срез работает.
- Этап закрыт только после merge в `master`, проверки `git status` и полного closeout.
- По окончании этапа нужно подвести итог и обновить документацию, технический долг и фиче-реквесты.
- Backend, тесты и установка Python-пакетов выполняются только в виртуальном окружении.
- Для Python-зависимостей используется `pip`.
- Код агента находится в `agent/`.
- MCP-эмулятор находится в `emulator/`.
- Frontend находится в папке `frontend/`.
- Seed-данные находятся в `datasets/`.
- Тесты разнесены по доменам в `tests/`.
- Локальный запуск без Docker должен работать на каждом этапе, где появляется исполняемый код.
- Docker добавляется как целевой способ упаковки, но не блокирует локальную разработку.
- Агент в MVP развивается по уровням: заглушка, простой исполнитель, агент с инструментами, агент с правками кода, агент с проверками и diff, агент с MCP tools, project profile и git-capability.
- Локальная БД является observability-хранилищем, а не источником бизнес-состояния агента.
- Перед управляющими решениями агент сверяется с live-состоянием таск-трекера, workspace и git.
- Архитектура должна допускать будущий централизованный сбор событий и управление несколькими агентами без переписывания runtime.

## Этап 2. MCP-Эмулятор Таск-Трекера

Статус: Реализован в ветке `stage-2-mcp-task-tracker-emulator`.

Результат: локально запускается MCP-эмулятор таск-трекера одного проекта.

Работы:

- Добавить загрузку состояния из указанного JSON-файла.
- Хранить состояние в памяти во время работы.
- При изменениях сбрасывать состояние во временный JSON-файл.
- Поддержать модель задачи:
  - идентификатор вида `PROJECT-123`;
  - тип `epic`, `task`, `test`;
  - статус из фиксированного workflow;
  - название;
  - автор email;
  - исполнитель email;
  - описание;
  - связи с другими задачами;
  - комментарии;
  - мета-информация как пары ключ-значение.
- Поддержать workflow `Todo`, `Open`, `InProgress`, `InReview`, `NeedsInfo`, `Done`, `Cancelled` с любыми переходами.
- Поддержать MCP-инструменты:
  - `workflow_get`;
  - `tasks_get`;
  - `tasks_list`;
  - `tasks_update`;
  - `comments_add`;
  - `comments_list`.
- Добавить seed JSON-файлы для локальных сценариев.

Проверка:

- MCP-эмулятор запускается с указанным JSON-файлом.
- Можно получить задачу по идентификатору.
- Можно получить список задач по статусу и исполнителю email.
- Можно изменить задачу, включая статус.
- Можно добавить комментарий.
- После изменения появляется snapshot JSON-файл с актуальным состоянием.

## Этап 3. Tick, MCP-Интеграция И Выбор Задачи

Статус: Реализован в ветке `stage-3-mcp-tick-task-selection`.

Результат: сервис агента может принять webhook или ручной tick, прочитать актуальные задачи из MCP-эмулятора, выбрать подходящую задачу и записать диагностический след выбора в observability-хранилище. Локальная БД не хранит задачи как источник правды.

Работы:

- Реализовать MCP-клиент-адаптер в агентской библиотеке.
- Зафиксировать, что прямой MCP-клиент является bootstrap-адаптером этапа 3, а целевая интеграция должна перейти в tool registry.
- Добавить настройки подключения к MCP-эмулятору и email агента.
- Добавить `POST /api/agent/tick` для ручного запуска выбора задачи.
- Добавить `POST /api/webhooks/task-tracker` как вход для webhook из таск-трекера.
- На tick читать workflow через `workflow_get`.
- Читать задачи через `tasks_list(status="Open", assignee_email=AGENT_EMAIL)`.
- Для кандидатов получать полные карточки через `tasks_get`.
- Проверять тип задачи, статус, исполнителя, приоритет и зависимости.
- Выбирать разблокированную задачу с максимальным приоритетом.
- Создавать `agent_tick`, `task_candidates`, события выбора и run для выбранной задачи.
- Обновить UI для ручного tick, просмотра кандидатов выбора и run.

Проверка:

- `POST /api/agent/tick` читает задачи из MCP-эмулятора.
- Сервис выбирает задачу с максимальным приоритетом среди разблокированных задач.
- Заблокированные задачи записываются в `task_candidates` как `skipped`.
- Если подходящих задач нет, tick завершается без run с понятным событием.
- UI запускает tick и показывает кандидатов выбора, выбранную задачу и run.

Работы:

  - `answer_only`;

  - `code_change`;
  - `blocked`.
- Добавить сохранение артефактов run в workspace.

- Добавить итоговый комментарий с типом результата, проверками и ссылкой на diff-артефакт, если он есть.
- Фиксировать, запускались ли проверки, или причину, почему проверки не запускались.
- Добавить seed-задачи, требующие правки кода.
- Обновить REST API и UI для просмотра артефактов.

Проверка:

- Agent может изменить файл в workspace.
- При файловых изменениях создается `final.diff`.
- Итоговый diff доступен через API и виден в UI.
- Задача с файловыми изменениями переводится в `InReview`, answer-only задача — в `Done`.

## Этап 8. Ослабление Зависимости От Локальной БД И Reconciliation

Статус: Реализован в ветке `stage-8-stateless-reconciliation`, смержен в `master`.

Результат: локальная SQLite-БД остается только observability-хранилищем. Выбор, старт, продолжение и завершение задачи опираются на live-состояние таск-трекера, workspace и git, а не на локальный `runs.status` как источник правды.

Работы:

- Зафиксировать `run` как диагностическую попытку выполнения, а не lock.
- Вынести правила claim/resume/skip/fail в отдельный policy layer.
- Перед стартом run перечитывать задачу через MCP и проверять статус, исполнителя, зависимости и признаки уже начатой работы.
- Заменить простой guard `queued/running` на reconciliation-логику:
  - если задача уже выполняется агентом и есть актуальный workspace/ветка, решить resume или skip;
  - если локальный run завис, но task tracker/git не подтверждают активную работу, записать диагностическое событие и разрешить новую попытку по policy;
  - если задача в неподходящем статусе, не стартовать и записать причину.
- Подготовить интерфейс observability sink, чтобы runtime не зависел напрямую от SQLite repository.
- Обновить REST/UI формулировки: показывать попытки выполнения и события, а не "истину" по состоянию задач.
- Обновить документацию по восстановлению и выбору задачи.

Проверка:

- Локальный зависший `running` run сам по себе не блокирует новую работу без подтверждения в таск-трекере/git.
- Две последовательные попытки по одной задаче оставляют понятный diagnostic trail.
- Поведение выбора покрыто тестами для live `Open`, live `InProgress`, отсутствующей ветки/workspace и устаревшего локального run.
- Сброс SQLite не ломает возможность выбрать и выполнить задачу, если таск-трекер и workspace находятся в согласованном состоянии.

## Этап 9. Рабочее Пространство Реального Репозитория И Git-Ветки

Статус: Реализован в ветке `stage-9-git-worktree-workspace`, смержен в `master`.

Результат: агент готовит рабочее пространство как реальный checkout проекта, создает или находит ветку задачи и может работать с diff на уровне git без обязательного создания коммита.

Работы:

- Добавить настройку `PROJECT_REPO_ROOT` для локального репозитория проекта.
- Для ручных и live smoke-прогонов вынести тестовый репозиторий в отдельную песочницу вне репозитория `simple_agent`.
- Зафиксировать стратегию workspace: `git worktree` на задачу.
- Создавать рабочую ветку вида `PROJECT-123-agent`.
- На старте задачи проверять существующую ветку с таким префиксом.
- Связать branch name с run как диагностическое поле.
- Оставить git commit выключенным до отдельной политики коммитов.

Проверка:

- Агент создает или находит ветку задачи.
- `git status` и `git diff` работают внутри подготовленного checkout.
- Локальный smoke запускается на отдельной песочнице и не создает ветки в репозитории `simple_agent`.
- Повторный tick по задаче видит существующую ветку и действует по reconciliation policy.
- Безопасность workspace сохраняется.

## Этап 12. Git Capability С Project-Defined Policy

Результат: агент может создавать git-коммиты, но решение о коммите, формат commit message и обязательные проверки определяются project profile, а не жесткой логикой runtime.

Работы:

- Добавить git tools для `git add`, `git commit`, просмотра последнего коммита и проверки чистоты дерева.
- Добавить project-defined commit policy.
- Добавить настройку, разрешающую или запрещающую коммиты для конкретного run.
- Требовать успешные проверки или явную policy-причину для пропуска.
- Публиковать hash созданного коммита в итоговом комментарии и observability.
- Не пушить изменения в `master` из автономных или фоновых запусков.

Проверка:

- Агент создает коммит в рабочей ветке тестового репозитория по project policy.
- Итоговый комментарий содержит hash коммита.
- Рабочее дерево после коммита чистое или причина отклонения явно записана.
- Автономный запуск не пушит в `master`.
