# План Реализации MVP

Цель плана — реализовывать систему постепенно, чтобы после каждого этапа оставался рабочий срез, который можно запустить локально и проверить.

## Принципы Выполнения

- Каждый этап выполняется в отдельной ветке.
- Перед началом этапа выполняется preflight по техническому долгу, инструментам и фиче-реквестам.
- Перед реализацией этапа нужен краткий план с декомпозицией и подтверждение.
- Этап считается завершенным только после локальной проверки и подтверждения.
- Merge в `master` выполняется после подтверждения, что срез работает.
- По окончании этапа нужно подвести итог и обновить документацию, технический долг и фиче-реквесты.
- Backend, тесты и установка Python-пакетов выполняются только в виртуальном окружении.
- Для Python-зависимостей используется `pip`.
- Frontend находится в папке `frontend/`.
- Локальный запуск без Docker должен работать на каждом этапе, где появляется исполняемый код.
- Docker добавляется как целевой способ упаковки, но не блокирует локальную разработку.
- Агент в MVP развивается по уровням: заглушка, простой исполнитель, агент с инструментами, агент с правками кода, агент с проверками и diff, агент с git-коммитами.

## Этап 0. Каркас Проекта И Инструменты Разработки

Статус: Реализован в ветке `stage-0-project-skeleton`.

Результат: проект можно установить в виртуальное окружение, запустить базовые проверки и увидеть healthcheck backend.

Работы:

- Создать Python-пакет `simple_agent`.
- Добавить `pyproject.toml`.
- Добавить установку зависимостей через `pip`.
- Добавить FastAPI-приложение с `GET /health`.
- Настроить pytest.
- Добавить базовую конфигурацию через переменные окружения.
- Добавить локальные инструкции запуска через активированное виртуальное окружение.
- Добавить минимальную структуру frontend в `frontend/` на React + Vite.

Проверка:

- Backend запускается локально.
- `GET /health` возвращает успешный ответ.
- Тесты запускаются из виртуального окружения.
- Frontend dev-server запускается из `frontend/` и показывает состояние backend healthcheck.

## Этап 1. Хранилище Наблюдаемости И Базовая Наблюдаемость

Статус: Реализован в ветке `stage-1-storage-observability`.

Результат: сервис умеет хранить диагностические tick-события, кандидатов выбора задач, будущие попытки выполнения задач агентом, события, вызовы инструментов и статистику без реального агента. Локальная БД не является источником правды по задачам.

Работы:

- Добавить SQLite-хранилище сервиса агента.
- Описать модели `agent_ticks`, `task_candidates`, `runs`, `events`, `tool_calls`.
- Реализовать repository layer.
- Добавить REST endpoints для чтения tick, кандидатов выбора, запусков, событий и статистики.
- Добавить тесты хранилища и API.

Проверка:

- Можно создать тестовые записи через код или фикстуры.
- REST API возвращает tick, кандидатов выбора, запуски, события и статистику.
- UI умеет показывать healthcheck, список tick, список запусков и статистику.

## Этап 2. MCP-Эмулятор Таск-Трекера

Результат: локально запускается MCP-эмулятор таск-трекера одного проекта.

Работы:

- Реализовать локальный MCP-сервер таск-трекера.
- Добавить загрузку состояния из указанного JSON-файла.
- Хранить состояние в памяти во время работы.
- При изменениях сбрасывать состояние во временный JSON-файл.
- Поддержать модель задачи:
  - идентификатор вида `PROJECT-123`;
  - тип `epic`, `task`, `test`;
  - статус из фиксированного workflow;
  - название;
  - автор email;
  - исполнитель email;
  - описание;
  - связи с другими задачами;
  - комментарии;
  - мета-информация как пары ключ-значение.
- Поддержать workflow `Todo`, `Open`, `InProgress`, `InReview`, `NeedsInfo`, `Done`, `Cancelled` с любыми переходами.
- Поддержать MCP-инструменты:
  - `workflow.get`;
  - `tasks.get`;
  - `tasks.list`;
  - `tasks.update`;
  - `comments.add`;
  - `comments.list`.
- Добавить seed JSON-файлы для локальных сценариев.

Проверка:

- MCP-эмулятор запускается с указанным JSON-файлом.
- Можно получить задачу по идентификатору.
- Можно получить список задач по статусу и исполнителю email.
- Можно изменить задачу, включая статус.
- Можно добавить комментарий.
- После изменения появляется snapshot JSON-файл с актуальным состоянием.

## Этап 3. Интеграция С MCP-Эмулятором И Синхронизация Задач

Результат: сервис агента может читать задачи из MCP-эмулятора и сохранять их в свое хранилище.

Работы:

- Реализовать MCP-клиент-адаптер в агентской библиотеке.
- Добавить `POST /tasks/sync`.
- Синхронизировать задачи по статусу и исполнителю email.
- Сохранять связь между внутренней задачей и внешним идентификатором вида `PROJECT-123`.
- Обновить UI для запуска синхронизации.

Проверка:

- `POST /tasks/sync` загружает задачи из MCP-эмулятора в хранилище сервиса.
- UI запускает синхронизацию и показывает импортированные задачи.

## Этап 4. Примитивный Агент Без LLM

Результат: можно запустить задачу, увидеть run, события и изменение статуса без LLM и без редактирования кода.

Работы:

- Реализовать машину состояний задачи.
- Реализовать `AgentController`.
- Реализовать runtime-заглушку, которая:
  - загружает задачу;
  - переводит ее в `InProgress`;
  - пишет комментарий о начале работы;
  - пишет комментарий о завершении;
  - переводит задачу в `Done`.
- Добавить endpoints:
  - `POST /tasks/{id}/run`;
  - `POST /tasks/{id}/cancel`;
  - `GET /runs/{id}/events`.
- Обновить UI для запуска и отмены задач.

Проверка:

- Задача запускается из REST API и UI.
- Создается run.
- События запуска видны через API и UI.
- Комментарии и статусы отражаются в MCP-эмуляторе.

## Этап 5. Агент С Рабочим Пространством И Примитивными Инструментами

Результат: агент может работать в изолированном рабочем пространстве и выполнять простые действия через инструменты.

Работы:

- Реализовать менеджер рабочих пространств.
- Добавить инструменты:
  - чтение файла;
  - запись файла;
  - patch файла;
  - список файлов;
  - поиск текста;
  - запуск команды с тайм-аутом;
  - `git status`;
  - `git diff`;
  - запуск тестов.
- Ограничить инструменты рамками рабочего пространства.
- Логировать вызовы инструментов в хранилище.
- Добавить тесты безопасности путей и тайм-аутов.
- Расширить примитивного агента, чтобы он мог выполнять заранее заданный сценарий инструментов без LLM.

Проверка:

- Агент создает рабочее пространство задачи.
- Инструменты работают внутри рабочего пространства.
- Попытка выйти за пределы рабочего пространства блокируется.
- Вызовы инструментов видны в истории run.

## Этап 6. LLM-Агент С Простым Циклом

Результат: агент может использовать LLM, составлять план, выбирать инструменты и завершать простые задачи.

Работы:

- Реализовать LLM-клиент с настройками `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL`.
- Добавить prompt layer для работы через задачи.
- Реализовать минимальный цикл:
  - анализ задачи;
  - план;
  - вызовы инструментов;
  - проверка результата;
  - итоговый комментарий.
- Добавить режим stub LLM для тестов.
- Ограничить количество шагов и вызовов инструментов на run.

Проверка:

- Агент выполняет простую локальную задачу на тестовом репозитории.
- Итоговый комментарий появляется в MCP-эмуляторе.
- Run содержит события LLM-вызовов и вызовов инструментов.
- Тесты не требуют реального LLM.

## Этап 7. Кодовый Агент С Проверками И Diff

Результат: агент может изменять код, запускать проверки и публиковать итоговый diff без создания git-коммита.

Работы:

- Разрешить агенту редактировать файлы в рабочем пространстве через инструменты.
- Добавить обязательный финальный `git diff`.
- Добавить обязательный запуск релевантных тестов или фиксированную причину, почему тесты не запускались.
- Добавить итоговый комментарий с кратким описанием изменений, проверками и ссылкой на diff-артефакт.
- Добавить сохранение diff как артефакта run.
- Добавить seed-задачи, требующие правки кода.

Проверка:

- Агент меняет код в тестовом репозитории.
- Агент запускает проверки.
- Итоговый diff доступен через API и виден в UI.
- Задача переводится в `InReview` или `Done` согласно настройке сценария.

## Этап 8. Git-Агент С Коммитами

Результат: агент может создавать git-коммит по результатам выполненной задачи.

Работы:

- Добавить инструменты:
  - `git add`;
  - `git commit`;
  - просмотр последнего коммита;
  - проверка чистоты рабочего дерева после коммита.
- Добавить политику commit message.
- Добавить настройку, разрешающую или запрещающую коммиты для конкретного run.
- Требовать успешные проверки или явную диагностическую причину перед коммитом.
- Публиковать hash созданного коммита в итоговом комментарии.
- Не пушить изменения в `master` из автономных или фоновых запусков.

Проверка:

- Агент создает коммит в рабочей ветке тестового репозитория.
- Итоговый комментарий содержит hash коммита.
- Рабочее дерево после коммита чистое или причина отклонения явно записана.
- Автономный запуск не пушит в `master`.

## Этап 9. Полный MVP-Сценарий

Результат: локально работает полный цикл от задачи в эмуляторе до выполнения агентом и просмотра результата в UI.

Работы:

- Подготовить демонстрационный репозиторий или fixture-проект.
- Добавить несколько seed-задач разных типов.
- Доработать UI:
  - список задач;
  - карточка задачи;
  - комментарии;
  - запуск и отмена;
  - события run;
  - diff-артефакты;
  - hash коммита, если он создан;
  - статистика;
  - healthcheck backend и MCP-эмулятора.
- Добавить документацию локального запуска.
- Добавить интеграционный тест полного цикла без реального LLM.

Проверка:

- Backend запускается локально.
- MCP-эмулятор запускается локально.
- UI запускается через Vite из `frontend/`.
- Пользователь может через UI синхронизировать задачи, запустить задачу и увидеть результат.

## Этап 10. Docker-Упаковка

Результат: сервис можно запустить в Docker, сохраняя возможность локальной разработки без Docker.

Работы:

- Добавить Dockerfile для backend.
- Добавить Dockerfile или сборочный режим для frontend.
- Добавить docker compose для backend, frontend и MCP-эмулятора.
- Настроить volume для SQLite, рабочих пространств и JSON-состояния MCP-эмулятора.
- Описать команды запуска.

Проверка:

- Полный MVP запускается через Docker.
- Локальный запуск без Docker продолжает работать.
