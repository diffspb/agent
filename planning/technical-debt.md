# Технический Долг

Этот файл отслеживает известные компромиссы и задачи на последующую очистку.

## Открытые Пункты

### TD-001: Определить Точное Соответствие Статусов Таск-Трекера

Статус: Открыт

В архитектуре используются общие статусы вроде `claimed`, `working`, `waiting_for_user` и `done`. Реальный workflow таск-трекера может использовать другие названия.

Нужно:

- Выбрать первый целевой таск-трекер.
- Сопоставить внутренние состояния со статусами трекера.
- Определить поведение при неуспешном переходе статуса.

### TD-002: Определить Политику Безопасности Для Инструментов

Статус: Открыт

Реестру инструментов нужна конкретная политика для выполнения shell-команд, ограничения путей, тайм-аутов и заблокированных операций.

Нужно:

- Выбрать тайм-аут по умолчанию для команд.
- Определить правила песочницы для путей.
- Определить denylist команд.
- Определить правила хранения артефактов.

### TD-003: Выбрать Слой Хранилища Для MVP

Статус: Открыт

Нужно:

- Определить стратегию миграций.
- Определить правила хранения событий и полезной нагрузки вызовов инструментов.

### TD-004: Описать Семантику Восстановления Запусков

Статус: Открыт

Нужно:

### TD-005: Зафиксировать Детальную Политику Git-Коммитов

Статус: Открыт

В MVP агент должен постепенно развиться до уровня, где он может создавать git-коммиты в рабочей ветке. Нужно детализировать точные правила создания коммитов.

Нужно:

- Определить формат commit message.
- Решить, какие проверки обязательны перед коммитом.
- Определить настройку, разрешающую или запрещающую коммиты для конкретного run.
- Решить, кто и когда переносит изменения из рабочей ветки в `master`.
- Зафиксировать формат итогового отчета по изменениям.

### TD-009: Добавить Vitest Для Frontend-Тестов

Статус: Открыт

Сейчас frontend этапа 0 проверяется через `npm run build` и smoke-проверку Vite dev-server. По мере роста UI нужно добавить unit/component tests на Vitest, чтобы не проверять базовую логику интерфейса через ручные HTTP smoke-команды.

Нужно:

- Добавить Vitest.
- Добавить Testing Library для React.
- Настроить `npm test`.
- Покрыть healthcheck UI: loading, success, error.
- Замокать `fetch` для проверки запросов к `/health`.

### TD-010: Разделить Регулярные Frontend-Тесты И Smoke-Проверки Dev-Server

Статус: Открыт

Smoke-проверка Vite dev-server через `curl` полезна для локальной валидации, но не должна быть основным способом проверки логики UI.

Нужно:

- Оставить `npm run build` как проверку сборки.
- Использовать `npm test` для логики UI после добавления Vitest.
- Оставить `curl --noproxy "*"` dev-server smoke как ручную или CI smoke-проверку.
- Обновить `docs/local-development.md` после внедрения Vitest.
