# Технический Долг

Этот файл отслеживает известные компромиссы и задачи на последующую очистку.

## Открытые Пункты

### TD-001: Определить Точное Соответствие Статусов Таск-Трекера

Статус: Открыт

В архитектуре используются общие статусы вроде `claimed`, `working`, `waiting_for_user` и `done`. Реальный workflow таск-трекера может использовать другие названия.

Нужно:

- Выбрать первый целевой таск-трекер.
- Сопоставить внутренние состояния со статусами трекера.
- Определить поведение при неуспешном переходе статуса.

### TD-004: Описать Семантику Reconciliation После Рестарта

Статус: Открыт

Runtime должен корректно вести себя после перезапуска, но восстановление не должно опираться только на локальный `runs.status`. Нужно сверять live-состояние задачи в таск-трекере, workspace и git.

Нужно:

- Решить, когда можно продолжить работу по существующему workspace/ветке.
- Решить, когда устаревший локальный `running` run нужно считать только диагностической записью.
- Решить, когда задача должна получать диагностический комментарий об ошибке после рестарта.
- Решить, как обнаруживать зависшую работу без превращения SQLite в источник правды.
- Добавить тесты reconciliation после рестарта процесса.

### TD-005: Зафиксировать Детальную Политику Git-Коммитов

Статус: Открыт

В MVP агент должен постепенно развиться до уровня, где он может создавать git-коммиты в рабочей ветке. При этом правила коммитов не должны быть жестко зашиты в runtime: они должны задаваться project profile и task context.

Нужно:

- Определить формат commit message.
- Решить, какие проверки обязательны перед коммитом.
- Определить настройку, разрешающую или запрещающую коммиты для конкретного run.
- Определить, какие части commit policy задаются глобально, а какие проектом.
- Решить, кто и когда переносит изменения из рабочей ветки в `master`.
- Зафиксировать формат итогового отчета по изменениям.
- Добавить тесты git capability перед включением автономных коммитов.

### TD-011: Добавить Стратегию Миграций SQLite-Схемы

Статус: Открыт

В этапе 1 схема SQLite создается через `CREATE TABLE IF NOT EXISTS`. Этого достаточно для первого MVP-среза, но при изменении схемы между этапами нужен явный подход к миграциям.

Нужно:

- Выбрать простой механизм версионирования схемы.
- Решить, допускается ли ручной сброс локальной базы в MVP.
- Описать порядок изменения таблиц и индексов.
- Добавить тест, который проверяет bootstrap новой схемы.

### TD-013: Добавить Пагинацию И Фильтры В Observability API

Статус: Открыт

С ростом количества tick, кандидатов, run и событий текущие endpoints наблюдаемости не должны возвращать неограниченные списки.

Нужно:

- Добавить query-параметры `limit` и `offset` или cursor-based пагинацию.
- Добавить фильтры по `external_task_id`, статусу run, source tick и периоду времени.
- Обновить UI для работы с постраничными списками.
- Добавить тесты на ограничения размера ответа.

### TD-014: Перевести MCP-Интеграции В Tool Registry

Статус: Открыт

В этапе 3 MCP-таск-трекер вызывается через детерминированный Python client adapter. Это допустимо для bootstrap/tick selection до появления LLM-agent runtime, но не должно стать целевой архитектурой агента. После live-прогонов стало ясно, что workflow, статусы и task-operations должны выполняться LLM через MCP tools, иначе runtime слишком жестко привязан к упрощенной модели трекера.

Нужно:

- Реализовать механизм подключения MCP tools в общий tool registry агента.
- Сохранять MCP tool calls в `tool_calls`.
- Передавать descriptions и schema MCP tools в prompt/tool layer.
- Убрать зависимость runtime от конкретных имен tools и жесткой workflow-логики task tracker.
- Оставить deterministic adapter только для bootstrap-сценариев выбора кандидата и preflight-checks.

### TD-015: Усилить Безопасность Инструментов До Production-Уровня

Статус: Открыт

В этапе 5 добавлена минимальная MVP-политика: workspace path sandbox, запуск shell без `shell=True`, denylist опасных команд, timeout и лимиты вывода. Этого достаточно для локального MVP, но недостаточно для production sandbox.

Нужно:

- Решить, нужен ли контейнерный запуск tools.
- Добавить allowlist команд для отдельных режимов.
- Ограничить ресурсы процесса: CPU, память, размер файлов.
- Добавить approvals для рискованных операций.
- Описать правила сетевого доступа tools.

### TD-017: Описать Retention И Cleanup Для Workspace-Артефактов

Статус: Открыт

В этапе 7 артефакты run сохраняются на диске в workspace и читаются через REST API. Для MVP этого достаточно, но не определены срок хранения, очистка старых workspace и поведение при переносе на Docker volume.

Нужно:

- Определить срок хранения workspace и артефактов.
- Добавить команду или задачу очистки старых run workspace.
- Решить, нужна ли индексная запись об артефактах в SQLite.
- Описать volume layout для Docker-режима.

### TD-018: Настроить Политику Финального Статуса Задачи По Workflow Проекта

Статус: Открыт

Сейчас runtime использует простую MVP-логику: `answer_only` переводит задачу в `Done`, а `code_change` — в `InReview`. В реальном проекте финальный статус зависит от workflow, типа задачи, правил ревью, политики проверок и настроек таск-трекера.

Нужно:

- Описать policy layer для выбора финального статуса задачи.
- Учитывать workflow таск-трекера и допустимые переходы.
- Поддержать настройки проекта для `answer_only`, `workspace_artifact`, `code_change` и `blocked`.
- Добавить тесты политики статусов после появления configurable workflow rules.

### TD-019: Добавить Полный Интеграционный Тест Backend + MCP Over HTTP

Статус: Открыт

Сейчас большая часть backend-тестов использует ASGI client и fake tracker. Это быстро и удобно, но не проверяет полный сетевой контур backend -> MCP emulator через HTTP.

Нужно:

- Запускать MCP-эмулятор на свободном порту в тесте.
- Запускать backend ASGI или uvicorn с `TASK_TRACKER_MCP_URL`.
- Проверять полный сценарий tick -> run start -> events -> artifacts.
- Учитывать WSL/proxy-настройки и обход proxy для локальных адресов.

### TD-020: Добавить E2E-Тесты UI

Статус: Открыт

Vitest покрывает компонентную логику UI, но не проверяет полный пользовательский сценарий в браузере.

Нужно:

- Добавить Playwright или аналогичный e2e-runner.
- Проверить сценарий tick -> start run -> events -> tool calls -> artifacts -> diff preview.

Нужно:

- Проверять workspace и git-ветку задачи.
- Записывать локальные расхождения как diagnostic events.
- Добавить тесты выбора задачи при устаревшем локальном `running` run.

### TD-023: Определить Стратегию Checkout/Worktree Для Реального Репозитория

Статус: Закрыт

Решение: в этапе 9 для MVP выбрана стратегия `git worktree` на задачу. `WorkspaceManager` поддерживает режим с `PROJECT_REPO_ROOT`, создает или переиспользует ветку вида `PROJECT-123-agent`, а `branch_name` записывается в run как observability-поле.
