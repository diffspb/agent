# Ограничения Разработки

Этот документ фиксирует обязательные ограничения для разработки MVP.

## Git Workflow

- Вся разработка функциональности ведется в отдельных ветках.
- Ветка мержится в `master` только после завершения этапа или задачи.
- Перед merge в `master` нужно подтверждение, что реализованный срез работает.
- В `master` можно пушить изменения документации и небольшие фиксы.
- Агентам, запущенным в фоне или автономном режиме, запрещено пушить в `master`.
- Автономный агент может готовить изменения в рабочей ветке, но перенос в `master` требует явного подтверждения.
- Если задача меняет поведение сервиса, перед merge должны быть пройдены релевантные тесты.

## Протокол Выполнения Этапа

Перед началом работы по этапу нужно выполнить preflight:

- Проверить открытые пункты технического долга и убедиться, что среди них нет блокеров для этапа.
- Проверить, что необходимые инструменты есть и работают.
- Если инструментов не хватает, убедиться, что их реализация входит в scope текущего этапа.
- Проверить фиче-реквесты и подсветить те, которые могут повлиять на текущий этап.
- Если фиче-реквест влияет на этап, сначала проработать его влияние и согласовать решение.

Если блокеров нет, перед началом реализации нужно кратко вывести план этапа:

- цель этапа;
- декомпозиция задач;
- затрагиваемые компоненты;
- ожидаемые проверки;
- ожидаемые изменения в документации.

К реализации можно приступать только после подтверждения плана.

Во время выполнения этапа:

- Вся работа ведется в ветке этапа.
- Документация обновляется вместе с кодом, если меняется поведение или архитектура.
- Новые ограничения, отложенные решения и риски фиксируются в `planning/technical-debt.md`.
- Идеи, которые не входят в текущий scope, фиксируются в `planning/feature-requests.md`.

По окончании этапа нужно подготовить итог:

- что удалось реализовать;
- что не получилось;
- какие проверки выполнены;
- появились ли новые пункты технического долга;
- появились ли новые фиче-реквесты;
- какие изменения внесены в документацию;
- нужно ли провести ревью тестов;
- нужно ли провести ревью архитектуры;
- нужно ли провести ревью процесса разработки.

Этап считается завершенным, когда:

- рабочий срез проверен;
- документация отражает актуальное состояние проекта;
- изменения смержены в `master` после подтверждения.

## Python И Виртуальное Окружение

- Основной backend-стек: Python.
- Все Python-зависимости устанавливаются только в виртуальное окружение.
- Все Python-тесты запускаются только из виртуального окружения.
- Для управления Python-зависимостями в MVP используется `pip`.
- Для активации окружения можно использовать алиас `vactivate` вместо `source .venv/bin/activate`.
- Команды установки и тестов в документации должны явно предполагать активированное виртуальное окружение.

## Локальный И Docker-Запуск

- Целевое использование сервиса — в Docker.
- Локальный запуск без Docker должен поддерживаться как основной сценарий разработки.
- Docker-конфигурация не должна быть обязательной для запуска тестов и локальной разработки.
- Локальный запуск должен позволять поднимать backend, тестовый веб-интерфейс и MCP-эмулятор таск-трекера.
- В WSL или окружениях с proxy локальные HTTP-проверки должны обходить proxy, например через `curl --noproxy "*"`.

## Frontend Для Тестирования REST API

- Нужен небольшой тестовый веб-интерфейс для выполнения команд REST API.
- Frontend делается на React.
- Frontend находится в папке `frontend/`.
- Сборка и dev-server frontend запускаются через Vite.
- UI является инструментом разработки и тестирования, а не полноценным продуктовым интерфейсом.
- UI должен покрывать основные команды: синхронизация задач, список задач, запуск задачи, отмена задачи, список запусков, события запуска, статистика и healthcheck.

## Эмулятор Таск-Трекера

- Для MVP нужен примитивный таск-трекер, эмулирующий реальную интеграцию.
- Взаимодействие агента с эмулятором должно идти через MCP.
- Эмулятор работает с задачами одного проекта.
- Эмулятор при запуске считывает состояние из указанного JSON-файла.
- Во время работы эмулятор хранит состояние в памяти.
- При изменении состояния эмулятор сбрасывает актуальное состояние во временный JSON-файл.
- Эмулятор должен поддерживать задачи, комментарии, статусы, связи задач и мета-информацию в минимальном виде.
- Эмулятор нужен для локальной разработки, тестов интеграции и демонстрации полного цикла.

## Запрещено В MVP

- Добавлять пользовательскую авторизацию.
- Делать полноценный production веб-интерфейс.
- Делать распределенную обработку задач.
- Требовать Docker для обычной локальной разработки.
- Запускать установку пакетов или тесты вне виртуального окружения.
- Пушить автономным или фоновым агентам напрямую в `master`.
