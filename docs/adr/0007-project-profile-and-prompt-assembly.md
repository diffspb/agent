# 0007. Project Profile И Многослойная Сборка Prompt

Дата: 2026-05-03
Статус: Принято

## Контекст

После live-прогонов с LLM стало видно, что жесткая логика workflow и git-поведения внутри runtime плохо переносится между проектами и таск-трекерами:

- реальные workflow заметно отличаются от упрощенного `Todo -> Open -> InProgress -> InReview -> Done`;
- правила работы с git зависят от проекта: формат веток, commit message, обязательные проверки, допустимость коммита и ревью;
- агент должен решать не только coding-задачи, но и задачи ревью, документации, архитектуры, исследования и task operations;
- один и тот же runtime должен уметь работать с разными проектами без переписывания Python-кода;
- агенту нужно передавать дополнительные знания о проекте и конкретной задаче.

## Решение

Ввести явный слой `project profile` и многослойную сборку prompt.

### 1. Разделить технический runtime mode и продуктовый рабочий режим

`AGENT_RUNTIME_MODE` остается технической настройкой движка:

- `primitive`
- `llm_stub`
- `llm`

Но продуктовые режимы работы агента не должны быть захардкожены в runtime. Они должны задаваться конфигурацией проекта и task context.

Примеры конфигурируемых режимов:

- реализация;
- ревью;
- документация;
- архитектура;
- исследование;
- операции с задачами;
- проектно-специфичные режимы.

### 2. Перенести workflow и task-operations в MCP tools

Runtime не должен знать конкретные статусы, имена переходов и прикладную семантику task tracker beyond bootstrap/tick selection.

Для LLM-цикла возможности task tracker должны быть представлены как MCP tools в общем tool registry. Это относится как минимум к:

- чтению задачи;
- поиску задач;
- чтению и добавлению комментариев;
- обновлению полей и статусов;
- чтению описания workflow;
- чтению связанных задач и метаданных.

### 3. Оставить runtime ответственным только за платформенные инварианты

Runtime отвечает за:

- lifecycle run;
- workspace;
- observability;
- лимиты шагов и tool calls;
- отмену и timeout;
- базовую reconciliation-логику;
- безопасную публикацию системной диагностики.

Решения вида:

- когда задавать вопрос;
- когда переводить задачу в review;
- когда коммитить;
- как формировать commit message;
- какие проверки обязательны;
- какой тип результата ожидается;

должны определяться project profile и prompt assembly.

### 4. Собирать prompt из нескольких слоев

Prompt должен состоять из независимых слоев:

1. платформенные системные инварианты;
2. project profile;
3. task context;
4. дополнительные knowledge attachments;
5. текущее состояние workspace и доступных tools.

Это должно позволить:

- ревьюить project behavior отдельно от Python-кода;
- хранить проектные правила в конфигурации и документах;
- постепенно добавлять новые task modes без переписывания runtime;
- использовать один и тот же runtime в разных проектах.

## Последствия

- Архитектура сдвигается от "runtime знает бизнес-правила" к "runtime дает безопасный каркас, а проект определяет поведение".
- Этап с git-коммитами должен идти после этапов с MCP tools и project profile.
- Появляется новый слой конфигурации и новый документ по prompt assembly.
- Технический долг `TD-014` становится частью ближайшего этапа, а политика git-коммитов (`TD-005`) переписывается как project-defined behavior.

## Рассмотренные Альтернативы

### Оставить workflow и git-политику в runtime

Отклонено. Это быстро для демо, но плохо переносится между проектами и трекерами.

### Перенести все поведение в prompt без platform runtime

Отклонено. Платформенные инварианты, безопасность, observability и lifecycle должны оставаться в коде.

### Захардкодить фиксированный список режимов агента

Отклонено. Набор режимов зависит от проекта и будет расширяться.
