# 0006. Stateless Бизнес-Поведение С Локальной Наблюдаемостью

Дата: 2026-05-03
Статус: Принято

## Контекст

Агент управляется через таск-трекер. Задачи, статусы, зависимости, вопросы, ответы и заметки агента должны жить в таск-трекере, а не в локальной базе агента. Изменения кода должны жить в git и рабочем пространстве задачи.

SQLite-хранилище появилось в MVP для истории запусков, событий, tool calls и UI-наблюдаемости. Если начать использовать это хранилище как источник управляющего состояния, агент станет сложнее восстанавливать после падений, а при появлении нескольких агентов возникнет риск рассинхронизации между локальными базами.

В будущем может появиться централизованное управление несколькими агентами и централизованный сбор событий. Это не входит в текущий MVP, но текущая архитектура не должна жестко связывать агентский цикл с локальным SQLite.

## Решение

Агент должен быть stateless по бизнес-логике и stateful только по наблюдаемости.

Источники правды:

- таск-трекер через MCP: задачи, статусы, зависимости, комментарии, вопросы, ответы и заметки агента;
- git и рабочее пространство: ветки, diff, коммиты, артефакты работы с кодом;
- LLM/tool loop: принимает решения на основе live-контекста задачи, workspace и доступных tools.

Локальное SQLite-хранилище:

- хранит диагностический след работы;
- хранит попытки выполнения как записи наблюдаемости, а не как lock или источник правды;
- используется REST API и тестовым UI для просмотра истории;
- не должно быть единственным основанием для выбора, блокировки, восстановления или завершения задачи.

Перед началом работы агент каждый раз должен читать live-состояние задачи через MCP и проверять зависимости. Если локальный `run` говорит одно, а таск-трекер и git показывают другое, управляющее решение должно опираться на таск-трекер/git, а локальная БД должна получить диагностическое событие о расхождении.

## Последствия

- Локальная БД может быть сброшена без потери бизнес-состояния задач.
- `runs.status` описывает состояние попытки выполнения, но не является распределенным lock.
- Восстановление после падения должно быть reconciliation-процессом по live-состоянию таск-трекера, git-ветке и workspace, а не простым продолжением локальной записи `running`.
- Для нескольких агентов понадобится внешний coordination/observability слой, но локальный агент уже будет к нему готов через явную границу событий.
- Observability-события нужно писать через узкий интерфейс, чтобы позже заменить SQLite на централизованный collector без переписывания runtime.
- Политики claim, resume, skip и final status должны быть отдельными policy-компонентами, а не неявной логикой repository.

## Рассмотренные Альтернативы

### SQLite Как Управляющее Состояние

Проще реализовать в одном процессе, но плохо масштабируется на несколько агентов и усложняет восстановление после падений. Отклонено как целевая архитектура.

### Полностью Stateless Без Локальной Истории

Самая чистая модель исполнения, но для MVP ухудшает отладку автономного агента и делает тестовый UI менее полезным. Отклонено для текущего MVP.

### JSONL-Файлы Вместо SQLite

Хорошо подходит для append-only логов и может быть полезно как будущий формат экспорта событий. Для текущего UI и статистики SQLite проще. Оставлено как возможный внутренний формат или транспорт для централизованной наблюдаемости.
