# 0005. LiteLLM Как Provider Layer Для LLM

Дата: 2026-05-02
Статус: Принято

## Контекст

Этап 6 добавляет LLM-цикл агента. Нужен способ вызывать разные модели и провайдеры, поддерживать tool calling и при этом не привязывать исполнительное ядро к конкретному API.

Писать собственный OpenAI-compatible HTTP-клиент можно, но это создаст лишний код для уже решенной задачи: авторизация, формат сообщений, tool calls, различия провайдеров и совместимость с локальными или hosted-моделями.

## Решение

Для MVP используется LiteLLM как тонкий provider layer.

В проекте остается собственный интерфейс `LLMClient`. Runtime зависит от этого интерфейса, а не от LiteLLM напрямую. Реализация `LiteLLMClient` отвечает только за вызов модели и нормализацию ответа во внутренние структуры.

Для тестов и локального сценария без реального токена используется `StubLLMClient`.

## Последствия

- Не нужно реализовывать низкоуровневый OpenAI-compatible клиент вручную.
- Можно подключать разные LLM-провайдеры через один слой.
- Агентный цикл, события, tool calls, workspace и task-tracker lifecycle остаются внутри проекта.
- Зависимость на LiteLLM добавляется в runtime-зависимости Python-пакета.
- Если LiteLLM перестанет подходить, замена будет ограничена реализацией `LLMClient`.

## Рассмотренные Альтернативы

- Собственный HTTP-клиент через `httpx`: проще по зависимостям, но дублирует provider logic и tool-call нюансы.
- Pydantic AI: сильный кандидат для будущего исследования, особенно из-за MCP-поддержки, но может начать дублировать наш runtime.
- OpenAI Agents SDK: подходит для OpenAI-oriented agent runtime, но сильнее влияет на архитектуру и provider neutrality.
- LangChain/LangGraph: мощный graph/runtime слой, но для текущего MVP выглядит тяжелее необходимого.
