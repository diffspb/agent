# 0004. SQLite-Хранилище Через Repository Layer

Дата: 2026-05-01
Статус: Принято

Обновление: роль SQLite уточнена в [ADR-0006](0006-stateless-business-behavior-with-local-observability.md). SQLite остается локальным хранилищем наблюдаемости и не должен становиться источником управляющего бизнес-состояния агента.

## Контекст

MVP нужен локальный слой хранения для наблюдаемости: webhook/tick-событий, кандидатов выбора задач, запусков, событий, вызовов инструментов и статистики. Хранилище должно работать локально без Docker и без внешних сервисов.

Таск-трекер остается источником правды по задачам, статусам, связям и комментариям. Git остается источником правды по веткам, коммитам и diff. Локальная БД не должна дублировать состояние задач как бизнес-источник.

## Решение

Для MVP используется SQLite-файл и стандартный модуль Python `sqlite3`.

Доступ к данным идет через `Repository`, а FastAPI-слой получает готовые методы чтения и записи. Схема создается при старте приложения через `CREATE TABLE IF NOT EXISTS`.

Путь к базе задается через `SIMPLE_AGENT_DB_PATH`, значение по умолчанию — `.data/simple-agent.sqlite3`.

## Последствия

- Локальный запуск не требует отдельной базы данных.
- Тесты могут использовать временные SQLite-файлы.
- Нет дополнительной runtime-зависимости на ORM.
- SQL остается внутри storage-слоя.
- Локальная БД может быть сброшена в MVP без потери источника правды по задачам.
- Перед несовместимым изменением схемы нужно добавить простую стратегию миграций или явно зафиксировать ручной сброс базы для MVP.
- Заметки агента, вопросы и пользовательски значимый прогресс должны писаться в комментарии таск-трекера, а не в локальную БД.

## Рассмотренные Альтернативы

### SQLAlchemy

Дает более богатую модель и миграционный путь, но добавляет зависимость и сложность раньше, чем это нужно MVP.

### SQLModel

Удобен для связки Pydantic и SQLAlchemy, но также добавляет ORM-слой и не нужен для текущего небольшого набора операций.

### Прямой JSON-Файл

Проще для старта, но хуже подходит для запросов, связей между задачами, запусками и событиями, а также для будущей статистики.
