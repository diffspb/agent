# Хранилище Наблюдаемости

## Назначение

Хранилище сервиса не является источником правды по задачам. Источники правды разделены так:

- таск-трекер хранит задачи, статусы, связи, комментарии, вопросы и заметки агента;
- git хранит фактическое состояние работы через ветки, коммиты и diff;
- SQLite-хранилище агента хранит только наблюдаемость: webhook/tick-события, попытки выполнения, кандидатов выбора, события, вызовы инструментов и агрегированную статистику.

Такой подход снижает риск рассинхронизации между локальной БД и таск-трекером. Агент каждый раз читает актуальное состояние задач через MCP, а в локальную БД пишет диагностический след своей работы.

## Реализация MVP

В MVP используется локальный SQLite-файл и стандартный модуль Python `sqlite3`.

Путь по умолчанию:

```text
.data/simple-agent.sqlite3
```

Путь можно изменить переменной окружения:

```text
SIMPLE_AGENT_DB_PATH
```

Схема создается при старте FastAPI-приложения через bootstrap `CREATE TABLE IF NOT EXISTS`. При несовместимых изменениях схемы в MVP допускается локальный сброс базы через `make reset-db`, потому что данные в ней являются логами и метриками, а не источником бизнес-состояния.

Переход со старой stage 1-схемы, где `runs` ссылалась на локальную таблицу `tasks`, выполняется автоматическим сбросом известных таблиц наблюдаемости. Это допустимо только для MVP и должно быть заменено явной миграционной стратегией перед появлением ценных исторических данных.

## Таблицы

### agent_ticks

Факт запуска обработки агентом. Источником может быть webhook, ручной REST-вызов, cron или будущий polling loop.

Основные поля:

- `id` — идентификатор tick.
- `source` — источник запуска, например `webhook` или `manual`.
- `status` — статус обработки tick.
- `trigger_task_id` — внешний идентификатор задачи, если webhook пришел по конкретной задаче.
- `payload_json` — исходный payload запуска.
- `started_at` — время начала обработки.
- `finished_at` — время завершения обработки.
- `error` — диагностическая ошибка.

### task_candidates

Снимок решения агента при выборе задачи. Запись показывает, какие задачи агент увидел, почему пропустил или выбрал конкретную задачу.

Основные поля:

- `id` — идентификатор записи.
- `tick_id` — ссылка на tick.
- `external_task_id` — идентификатор задачи в таск-трекере, например `PROJECT-123`.
- `status` — статус задачи в таск-трекере на момент выбора.
- `assignee_email` — исполнитель.
- `priority` — числовой приоритет, если он известен.
- `dependencies_state` — состояние зависимостей: `clear`, `blocked`, `unknown`.
- `decision` — `selected` или `skipped`.
- `reason` — причина выбора или пропуска.
- `metadata_json` — дополнительная диагностическая информация.

### runs

Одна попытка выполнения внешней задачи агентом.

Основные поля:

- `id` — идентификатор запуска.
- `tick_id` — tick, в рамках которого выбран запуск.
- `external_task_id` — идентификатор задачи в таск-трекере.
- `branch_name` — git-ветка, связанная с задачей.
- `status` — статус запуска.
- `started_at` — время старта.
- `finished_at` — время завершения.
- `summary` — краткий технический итог.
- `error` — диагностическая ошибка.

### events

Лента событий tick или run.

Основные поля:

- `id` — идентификатор события.
- `tick_id` — ссылка на tick, если событие относится к выбору/обработке tick.
- `run_id` — ссылка на run, если событие относится к выполнению задачи.
- `type` — тип события, например `tick.started`, `task.selected`, `llm.requested`, `tool.completed`, `run.completed`.
- `message` — короткое сообщение.
- `payload_json` — структурированная диагностическая нагрузка.

### tool_calls

История вызовов инструментов агентом.

Основные поля:

- `id` — идентификатор вызова.
- `run_id` — ссылка на run.
- `tool_name` — имя инструмента.
- `status` — статус вызова.
- `input_json` — сокращенный вход инструмента.
- `output_json` — сокращенный результат инструмента.
- `error` — ошибка.
- `started_at` — время старта.
- `finished_at` — время завершения.

Большие выводы команд, diff, логи и артефакты не должны храниться целиком в БД. В БД хранится summary, путь к артефакту или hash.

## Repository Layer

Код сервиса не работает с SQLite напрямую. Доступ идет через `Repository`, чтобы:

- не смешивать SQL с HTTP-слоем;
- упростить тестирование;
- централизовать сериализацию JSON-полей и дат;
- удерживать границу между наблюдаемостью и источниками правды.

## REST API Наблюдаемости

```text
GET /api/ticks
GET /api/ticks/{id}
GET /api/ticks/{id}/candidates
GET /api/runs
GET /api/runs/{id}
GET /api/runs/{id}/events
GET /api/runs/{id}/tool-calls
GET /api/stats
```

Эти endpoints читают диагностическое состояние агента. Список актуальных задач должен читаться live из MCP-таск-трекера на следующих этапах, а не из локальной SQLite-БД.
