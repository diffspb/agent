# Хранилище Сервиса

## Назначение

Хранилище сервиса хранит внутреннее состояние агента: локальные задачи, попытки выполнения, события, вызовы инструментов, заметки агента и агрегированную статистику.

Таск-трекер остается источником пользовательской задачи и коммуникации. Хранилище сервиса отвечает за техническую историю выполнения и наблюдаемость.

## Реализация MVP

В этапе 1 используется локальный SQLite-файл и стандартный модуль Python `sqlite3`.

Путь по умолчанию:

```text
.data/simple-agent.sqlite3
```

Путь можно изменить переменной окружения:

```text
SIMPLE_AGENT_DB_PATH
```

Схема создается при старте FastAPI-приложения через bootstrap `CREATE TABLE IF NOT EXISTS`. Полноценная система миграций отложена до момента, когда схема начнет меняться между версиями.

## Таблицы

### tasks

Локальная копия задачи или внутренняя задача сервиса.

Основные поля:

- `id` — внутренний числовой идентификатор.
- `external_id` — идентификатор во внешнем таск-трекере, например `PROJECT-123`.
- `type` — тип задачи.
- `status` — текущий статус.
- `title` — название.
- `author_email` — автор.
- `assignee_email` — исполнитель.
- `description` — описание.
- `metadata_json` — произвольные метаданные.

### runs

Одна попытка выполнения задачи агентом.

Основные поля:

- `id` — идентификатор запуска.
- `task_id` — ссылка на локальную задачу.
- `status` — статус запуска.
- `started_at` — время старта.
- `finished_at` — время завершения.
- `summary` — краткий итог.
- `error` — диагностическая ошибка.

### events

Лента событий запуска.

Основные поля:

- `id` — идентификатор события.
- `run_id` — ссылка на запуск.
- `type` — тип события.
- `message` — короткое сообщение.
- `payload_json` — структурированная нагрузка события.

### tool_calls

История вызовов инструментов агентом. На этапе 1 таблица создается, но реальные записи появятся на этапах с инструментами.

### agent_notes

Внутренние заметки агента. На этапе 1 таблица создается, но реальные записи появятся на этапах с runtime агента.

## Repository Layer

Код сервиса не работает с SQLite напрямую. Доступ идет через `Repository`, чтобы:

- не смешивать SQL с HTTP-слоем;
- упростить тестирование;
- оставить возможность заменить реализацию хранилища позже;
- централизовать сериализацию JSON-полей и дат.

## REST API Этапа 1

```text
GET /api/tasks
GET /api/tasks/{id}
GET /api/runs
GET /api/runs/{id}
GET /api/runs/{id}/events
GET /api/stats
```

Эти endpoints только читают данные. Запись тестовых данных выполняется через repository layer в тестах. Реальные записи появятся на следующих этапах: синхронизация с MCP, запуск задач и вызовы инструментов.
