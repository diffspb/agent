# Выбор Задачи Агентом

## Назначение

Этот документ фиксирует MVP-логику выбора задачи после webhook или ручного tick. Агент не хранит задачи локально как источник правды. На каждом tick он читает актуальное состояние задач из таск-трекера через MCP, принимает решение и записывает в SQLite только диагностический след выбора.

## Источники Данных

Источник правды по задачам:

- MCP-таск-трекер или MCP-эмулятор;
- задача, ее статус, исполнитель, связи, комментарии и metadata;
- workflow таск-трекера.

Источник правды по работе с кодом:

- git-ветки;
- diff;
- коммиты на поздних этапах MVP.

Хранилище агента:

- `agent_ticks`: факт запуска выбора/обработки;
- `task_candidates`: задачи, которые агент рассмотрел, и причины выбора или пропуска;
- `runs`: попытка выполнения выбранной внешней задачи;
- `events`: события tick или run;
- `tool_calls`: вызовы инструментов на следующих этапах.

## Tick

Tick — это одна попытка агента найти подходящую задачу и, если возможно, начать работу.

Источники tick:

- webhook из таск-трекера;
- ручной REST-вызов;
- будущий polling loop.

На старте tick агент пишет `agent_ticks` со статусом `started`, source и исходным payload. После выбора и запуска задачи tick завершается статусом `completed`. Если подходящей задачи нет, tick тоже завершается `completed`, а причина фиксируется в событиях и `task_candidates`. Если произошла ошибка инфраструктуры, tick завершается `failed`.

## Базовые Фильтры

Для MVP агент рассматривает только задачи:

- `type` равен `task` или `test`;
- `status` равен `Open`;
- `assignee_email` равен email агента;
- задача не заблокирована зависимостями.

Статусы вне выбора:

- `Todo`: не выбирается автоматически. Это задача, которую потенциально нужно делать, но она еще не открыта для выполнения: может быть плохо сформулирована или еще не принято решение о выполнении.
- `InProgress`: не выбирается как новая задача; позже может использоваться для восстановления run.
- `InReview`: не выбирается для выполнения.
- `NeedsInfo`: не выбирается; агент должен ждать ответа в комментариях.
- `Done`: не выбирается.
- `Cancelled`: не выбирается.

Email агента должен быть настройкой сервиса, например `AGENT_EMAIL`.

## Зависимости

Задача считается заблокированной, если выполняется хотя бы одно условие:

- у нее есть link `blocked_by` на задачу, статус которой не `Done` и не `Cancelled`;
- другая задача имеет link `blocks` на текущую задачу, и статус этой другой задачи не `Done` и не `Cancelled`.

Задача считается разблокированной, если:

- блокирующих задач нет;
- все блокирующие задачи находятся в `Done` или `Cancelled`.

`Cancelled` считается разблокирующим статусом, потому что отмененная задача фактически больше не существует как работа и не должна блокировать другие задачи.

Если связанная задача не найдена через MCP, состояние зависимостей считается `unknown`, а кандидат пропускается. Это консервативное правило для MVP: лучше не начать работу, чем проигнорировать возможный блокер.

`part_of_epic` не блокирует задачу. Эта связь используется только как контекст и может попадать в metadata кандидата.

## Приоритет

В MVP приоритет читается из универсального поля `metadata.priority`. Отдельное поле `metadata.priority_score` не вводится.

Поддерживаемые значения:

```text
critical -> 400
high     -> 300
normal   -> 200
low      -> 100
```

Если `metadata.priority` отсутствует или неизвестен, используется `normal`.

Если `metadata.priority` содержит число или строку с числом, агент использует это число напрямую. Это позволяет seed-файлам и будущим интеграциям задавать более точный порядок.

## Алгоритм Выбора

1. Создать `agent_tick`.
2. Получить workflow через `workflow_get`.
3. Получить задачи через `tasks_list(status="Open", assignee_email=AGENT_EMAIL)`.
4. Для каждой задачи получить полную карточку через `tasks_get`.
5. Проверить тип задачи.
6. Проверить зависимости через links текущей задачи и связанные задачи.
7. Рассчитать priority score.
8. Записать каждого кандидата в `task_candidates`:
   - `selected`, если задача выбрана;
   - `skipped`, если задача не подходит;
   - reason с конкретной причиной.
9. Выбрать задачу с максимальным priority score среди разблокированных.
10. При равном score выбрать задачу с меньшим числовым номером в идентификаторе, например `PROJECT-7` раньше `PROJECT-12`.
11. Создать `run` для выбранной задачи.
12. Записать события выбора в `events`.

## Причины Пропуска

Минимальные reason-коды для `task_candidates`:

```text
status_not_open
wrong_assignee
unsupported_task_type
blocked_by_dependency
dependency_unknown
lower_priority_than_selected
already_running
```

Если задача выбрана:

```text
selected_highest_priority
```

## Git-Ветка

На этапе выбора задачи агент не должен полагаться на локальную БД. Для проверки текущей работы используется git:

- имя рабочей ветки должно начинаться с внешнего номера задачи, например `PROJECT-123-...`;
- если такая ветка уже есть, агент может использовать ее как признак ранее начатой работы;
- если активный run уже существует для задачи, новая попытка не стартует без отдельного правила восстановления.

Детальная политика восстановления run остается в `TD-004`.

## Наблюдаемость

Агент пишет в SQLite:

- tick source и payload;
- список кандидатов;
- причину выбора/пропуска;
- выбранный `external_task_id`;
- рассчитанный priority score;
- состояние зависимостей;
- события выбора и ошибок.

Агент не пишет в SQLite:

- полную копию задачи как источник правды;
- комментарии задачи как рабочую память;
- долгосрочные заметки по задаче.

Вопросы, заметки и пользовательски значимый прогресс пишутся в комментарии таск-трекера.

## Остаточные Риски Из Архитектурного Ревью

Закрыто изменением подхода к хранению:

- локальная БД больше не должна синхронизировать и хранить задачи как source of truth;
- риск рассинхронизации локальной таблицы `tasks` с таск-трекером снят;
- `external_updated_at`, `last_synced_at` и `raw_payload` для локальных задач не нужны.

Осталось актуально:

- нужна политика лимитов и редактирования payload/tool output, см. `TD-002`;
- нужна стратегия миграций SQLite-схемы, см. `TD-011`;
- нужна семантика восстановления незавершенных запусков, см. `TD-004`;
- нужна пагинация и фильтры в observability API перед ростом количества событий;
- нужен маппинг статусов для реального таск-трекера, см. `TD-001`.

## Подтвержденные Решения Для Этапа 3

- `Todo` не попадает в авто-выбор, агент выбирает только `Open`.
- `Cancelled` блокирующей задачи разблокирует зависимую задачу наравне с `Done`.
- Для приоритета используется универсальное поле `metadata.priority`.
