# Архитектура

## Назначение

Система представляет собой автономного агента для написания кода, управляемого через таск-трекер. Таск-трекер является основным пользовательским интерфейсом: требования приходят в виде задач, вопросы задаются в комментариях, а прогресс отражается через статусы.

Сам агент должен быть реализован как Python-библиотека. FastAPI-сервис оборачивает эту библиотеку и предоставляет операционные эндпоинты для запуска работы, отмены работы, чтения истории попыток выполнения и статистики. Актуальное состояние задач читается из таск-трекера через MCP, а не из локальной базы агента.

## Объем MVP

Входит в MVP:

- Автономный цикл выполнения задач.
- Интеграция с таск-трекером через MCP.
- Примитивный MCP-эмулятор таск-трекера для локальной разработки.
- Управляющее API на FastAPI.
- Тестовый веб-интерфейс на React и Vite для выполнения команд REST API.
- Локальный реестр инструментов для работы с кодом.
- Локальная наблюдаемость для попыток выполнения, событий, вызовов инструментов и статистики.
- Управление изолированными рабочими пространствами.
- Комментарии к задачам для прогресса, вопросов и итоговых отчетов.

Не входит в MVP:

- Авторизация пользователей и управление ролями.
- Сложный веб-интерфейс.
- Распределенные рабочие процессы.
- Развертывание в Kubernetes.
- Оркестрация нескольких агентов.
- Долгосрочная векторная память.
- Ручное подтверждение каждого вызова инструмента.
- Полная автоматизация браузера.

## Основные Компоненты

### Исполнительное Ядро Агента

Исполнительное ядро, или runtime, — ядро библиотеки. Оно управляет жизненным циклом одной попытки выполнения задачи:

```text
найти задачу
-> загрузить контекст задачи
-> составить план выполнения
-> вызывать инструменты
-> редактировать код
-> запускать проверки
-> обновлять задачу и комментарии
-> завершить работу или задать вопрос
```

Ответственность:

- Координировать lifecycle попытки выполнения.
- Создавать и обновлять план выполнения.
- Выбирать и вызывать инструменты.
- Передавать события, решения и результаты инструментов в слой наблюдаемости.
- Сверять live-состояние задачи с таск-трекером перед управляющими действиями.
- Восстанавливаться после прерванных запусков через reconciliation по таск-трекеру, git и workspace.
- Задавать уточняющие вопросы через комментарии.
- Формировать итоговые отчеты по задаче.

В этапе 4 реализован `PrimitiveAgentRuntime` без LLM и без редактирования кода. Он запускает уже созданный run, загружает задачу из MCP-трекера, переводит ее в `InProgress`, пишет комментарии о начале и завершении stub-работы, переводит задачу в `Done` и завершает run. Это первый исполняемый контур жизненного цикла, но не целевой кодовый агент.

В этапе 6 добавлен `LLMAgentRuntime`. Он сохраняет тот же lifecycle задачи, но внутри выполнения использует `LLMClient`, prompt layer и локальный registry инструментов. Доступны режимы `primitive`, `llm_stub` и `llm`, выбираемые через `AGENT_RUNTIME_MODE`.

В этапе 7 общий lifecycle вынесен в `TaskLifecycleRuntime`. Конкретные runtime-стратегии реализуют только `execute`: primitive stub, LLM loop и будущие code/diff/git-режимы. Runtime формирует outcome результата:

- `answer_only` — достаточно итогового комментария, файловых изменений нет;
- `workspace_artifact` — создан артефакт без трактовки результата как правки кода;
- `code_change` — изменены файлы в workspace, создан `final.diff`, задача переводится в `InReview`;
- `blocked` — выполнение остановлено из-за нехватки информации или внешнего блокера.

Git не является обязательной частью lifecycle. Diff этапа 7 строится по снимкам файлов workspace, а git остается отдельным набором tools и политик для следующих этапов.

Runtime не должен считать локальный `runs.status` источником правды по задаче. Это диагностическое состояние попытки. Управляющие решения принимаются по live-состоянию таск-трекера, workspace и git.

### MCP-Клиент Таск-Трекера

Доступ к таск-трекеру должен идти через адаптер, а не напрямую из runtime. Адаптер скрывает специфичные для трекера имена MCP-инструментов и формы данных за стабильным интерфейсом проекта.

В этапе 3 прямой MCP-клиент используется только bootstrap-сервисом выбора задачи. Целевой runtime должен получать MCP-возможности через общий реестр инструментов, чтобы агент мог работать с разными MCP-трекерами без жесткой привязки к конкретным именам tools. Переход зафиксирован в `TD-014`.

Минимальный интерфейс:

```python
class TaskTracker:
    async def get_workflow(self) -> Workflow: ...
    async def list_tasks(self, status: str, assignee_email: str) -> list[Task]: ...
    async def get_task(self, task_id: str) -> TaskDetails: ...
    async def update_task(self, task_id: str, patch: dict) -> TaskDetails: ...
    async def list_comments(self, task_id: str) -> list[Comment]: ...
    async def add_comment(self, task_id: str, author_email: str, text: str) -> None: ...
```

Полезные MCP-инструменты для трекера:

- `workflow_get`
- `tasks_get`
- `tasks_list`
- `tasks_update`
- `comments_add`
- `comments_list`

MCP-ресурсы могут отдавать описания задач, вложения, критерии приемки, метаданные репозитория и связанные документы.

### MCP-Эмулятор Таск-Трекера

Для MVP нужен локальный MCP-эмулятор таск-трекера. Он позволяет разрабатывать агентский цикл без подключения к реальному внешнему трекеру.

Ответственность:

- Работать как трекер задач одного проекта.
- При запуске читать состояние из указанного JSON-файла.
- Во время работы хранить состояние в памяти.
- При изменениях сбрасывать актуальное состояние во временный JSON-файл.
- Хранить задачи, комментарии, связи задач, мета-информацию и статусы.
- Отдавать задачи через MCP-инструменты.
- Принимать комментарии агента.
- Обновлять статусы задач.
- Поддерживать seed-данные для локального запуска.
- Быть достаточно простым, чтобы его можно было использовать в интеграционных тестах.

Подробная модель описана в [документе MCP-эмулятора таск-трекера](mcp-task-tracker-emulator.md).

### LLM-Клиент

LLM-клиент должен быть нейтральным к провайдеру. MVP может ориентироваться на API, совместимое с OpenAI, но runtime должен зависеть от проектного интерфейса.

```python
class LLMClient:
    async def complete(self, messages, tools=None, model=None) -> LLMResponse:
        ...
```

Ожидаемая конфигурация:

```text
LLM_BASE_URL
LLM_API_KEY
LLM_MODEL
LLM_MAX_STEPS
LLM_TIMEOUT_SECONDS
```

В этапе 6 в качестве provider layer выбран LiteLLM. Runtime зависит от собственного интерфейса `LLMClient`; `LiteLLMClient` только нормализует вызовы и ответы LiteLLM. Решение зафиксировано в [ADR-0005](adr/0005-litellm-provider-layer.md).

### Реестр Инструментов

Реестр инструментов предоставляет агенту контролируемые возможности. Инструменты должны быть типизированы, логироваться, иметь тайм-аут и быть ограничены рабочим пространством задачи.

Группы инструментов для MVP:

Файловая система:

- `read_file`
- `write_file`
- `patch_file`
- `list_files`
- `search_text`
- `inspect_tree`

Выполнение кода:

- `run_command`
- `run_tests`
- `run_linter`
- `run_formatter`

Git:

- `git_status`
- `git_diff`
- `git_show`
- `git_branch`
- `git_add`
- `git_commit`

Таск-трекер:

- `read_task`
- `post_comment`
- `ask_question`
- `set_task_status`
- `attach_report`

Контроль безопасности:

- Список запретов для очевидно разрушительных shell-команд.
- Тайм-аут для команд.
- Ограничение размера читаемых файлов.
- Ограничение путей рамками рабочего пространства.
- Запись каждого вызова инструмента и краткого результата.

### Менеджер Рабочих Пространств

Каждой задаче нужно контролируемое рабочее пространство.

```text
workspaces/
  task-123/
    repo/
    artifacts/
    logs/
    scratch/
```

Ответственность:

- Создавать и находить рабочие пространства задач.
- Клонировать или готовить репозитории.
- Создавать ветки под задачи, если это нужно.
- Хранить логи и артефакты.
- Гарантировать, что инструменты не работают за пределами рабочего пространства.

Артефакты run хранятся в каталоге `artifacts/` конкретного workspace и доступны через REST API. SQLite хранит события и tool calls, но не является хранилищем больших файлов, diff и логов.

Для MVP достаточно локальной изоляции через файловую систему. Контейнеры можно добавить позже.

### Хранилище Наблюдаемости

Сервису нужен диагностический след для UI, статистики, отладки и анализа качества работы агента. Локальная БД не должна становиться источником правды по задачам, lock-механизмом или обязательной рабочей памятью агента.

Источники правды:

- задачи, статусы, зависимости, вопросы и заметки агента — таск-трекер через MCP;
- изменения кода — git, workspace, diff и коммиты;
- история наблюдаемости — локальный SQLite в MVP, с возможностью последующей замены на централизованный collector.

В этапе 1 хранилище реализовано как локальный SQLite-файл через стандартный модуль Python `sqlite3`. Оно хранит webhook/tick-события, попытки выполнения, кандидатов выбора, события и вызовы инструментов. Подробности схемы, ограничения и REST API описаны в [документе хранилища](storage.md). Архитектурное решение о stateless бизнес-поведении зафиксировано в [ADR-0006](adr/0006-stateless-business-behavior-with-local-observability.md).

### FastAPI-Сервис

FastAPI — управляющий слой. Он должен оставаться тонким и делегировать реальную работу агентской библиотеке.

Минимальные эндпоинты:

```text
POST /api/webhooks/task-tracker
POST /api/agent/tick

GET  /api/ticks
GET  /api/ticks/{id}
GET  /api/ticks/{id}/candidates
GET  /api/runs
GET  /api/runs/{id}
GET  /api/runs/{id}/events

GET  /api/stats
GET  /health
```

Опциональные операционные эндпоинты:

```text
POST /agent/start
POST /agent/stop
GET  /agent/status
```

### Тестовый Веб-Интерфейс

Тестовый веб-интерфейс нужен для ручной проверки REST API и демонстрации полного сценария. Он не является полноценным продуктовым интерфейсом.

Технологии:

- React.
- Vite.

Минимальные экраны и действия:

- Проверка `GET /health`.
- Запуск ручного tick через REST API.
- Список последних tick.
- Кандидаты выбора задачи для tick.
- Список запусков.
- События запуска.
- Статистика агента.

### Контроллер Агента

Контроллер координирует фоновое выполнение и связывает FastAPI с runtime.

Ответственность:

- Запускать и останавливать фоновый опрос.
- Запускать конкретную задачу.
- Отменять выполняющуюся задачу.
- Ограничивать параллелизм.
- Не допускать дублирующую работу по одной задаче через live-проверки таск-трекера, workspace и git.
- Выполнять reconciliation после рестарта без превращения локального `runs.status` в источник правды.

Предлагаемый интерфейс:

```python
class AgentController:
    async def start(self): ...
    async def stop(self): ...
    async def handle_tick(self): ...
    async def run_next_task(self): ...
    async def cancel_run(self, run_id: int): ...
```

Выбор задачи описан отдельно в [документе выбора задачи](task-selection.md).

### Слой Промптов И Политик

Промпты и политики нужно рассматривать как версионируемое поведение проекта.

Системный prompt должен требовать от агента:

- Работать только через задачи.
- Задавать вопросы только через комментарии к задачам.
- Переводить заблокированные задачи в `waiting_for_user`.
- Избегать несвязанных рефакторингов.
- Запускать проверки перед завершением задачи.
- Публиковать итоговые отчеты через комментарии.
- Оставлять видимую диагностику при ошибках.
- Оставаться в рамках рабочего пространства.

## Жизненный Цикл Выполнения

```text
1. Получить webhook или ручной tick.
2. Создать `agent_tick` в хранилище наблюдаемости.
3. Через MCP получить workflow и список задач-кандидатов.
4. Проверить статусы, исполнителя, приоритет и зависимости.
5. Записать кандидатов и причины выбора/пропуска в `task_candidates`.
6. Если подходящей задачи нет, завершить tick без run.
7. Если задача выбрана, создать diagnostic run-attempt по `external_task_id`.
8. Загрузить описание, поля, комментарии, вложения и связанные ресурсы.
9. Подготовить рабочее пространство.
10. Проанализировать репозиторий и релевантные файлы.
11. Составить короткий план выполнения.
12. Выполнить работу через инструменты.
13. Запустить проверки и доработать решение.
14. Если есть блокер, задать вопрос в комментарии и остановить запуск.
15. Если задача выполнена, опубликовать итоговый отчет, приложить артефакты и обновить статус.
16. Если произошла ошибка, опубликовать диагностику в комментарий задачи и записать событие наблюдаемости.
```
