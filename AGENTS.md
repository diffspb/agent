# AGENTS.md

## Цель Проекта

Нужно разработать автономного агента для написания кода и выполнения смежных инженерных задач. Основной интерфейс взаимодействия с агентом — таск-трекер: требования передаются через задачи, уточнения обсуждаются в комментариях, прогресс отражается через статусы задач.

Агент должен быть реализован как Python-библиотека. Поверх библиотеки должен быть FastAPI-сервис, который предоставляет управляющие эндпоинты для инициации задач, просмотра статуса, истории запусков и статистики работы агента.

## Базовые Вводные

- В MVP не нужна авторизация пользователей и управление ролями.
- Доступ к LLM настраивается через URL, API-токен и имя модели.
- Таск-трекер подключается через MCP.
- REST API нужен для управления и наблюдаемости, а не для диалогового общения.
- Все пользовательское взаимодействие по задаче идет через описание задачи, поля задачи и комментарии.
- Агент должен автономно выполнять задачи по коду и смежные инженерные задачи.
- Агент должен сохранять достаточно состояния, чтобы восстанавливаться после перезапуска процесса.
- Исполнительное ядро агента должно быть независимым от FastAPI.

## Возможности MVP

- Находить или принимать задачи.
- Захватывать задачу перед началом работы.
- Загружать контекст задачи из таск-трекера через MCP.
- Готовить изолированное рабочее пространство для задачи.
- Анализировать код, редактировать файлы, запускать команды, тесты, линтеры и форматтеры.
- Задавать вопросы через комментарии к задаче и переводить задачу в состояние ожидания.
- Публиковать прогресс и итоговый отчет через комментарии к задаче.
- Сохранять запуски, вызовы инструментов, события и базовую статистику.
- Предоставлять REST-эндпоинты для запуска работы, отмены работы, чтения статуса и статистики.

## Предпочтительная Архитектура

- Python-пакет для исполнительного ядра агента.
- FastAPI-приложение как тонкий управляющий слой.
- MCP-клиент-адаптер для таск-трекера.
- Локальный реестр инструментов для файловой системы, shell, git, тестов и действий в таск-трекере.
- Менеджер рабочих пространств с отдельным рабочим пространством на задачу или checkout репозитория.
- Хранилище состояния на SQLite для MVP, с возможностью перехода на PostgreSQL позже.
- Явная машина состояний для выполнения задач.

## Инженерные Правила

- Исполнительное ядро должно оставаться пригодным для использования как библиотека без FastAPI.
- Интеграции нужно прятать за интерфейсами.
- Предпочитать небольшие заменяемые адаптеры вместо жесткой привязки к конкретным внешним системам.
- Не добавлять авторизацию, многопользовательские права, Kubernetes или распределенные рабочие процессы в MVP.
- Не скрывать ошибки от таск-трекера: неуспешный запуск должен оставлять диагностический комментарий.
- Не считать, что LLM-провайдер обязательно OpenAI; нужна адаптируемая клиентская абстракция.
- Комментарии к задаче являются частью продуктового поведения, а не отладочным журналом.

## Границы Первой Реализации

Первая реализация должна сфокусироваться на:

1. Исполнительном ядре агента и машине состояний.
2. Интерфейсе MCP-клиента для таск-трекера.
3. Реестре инструментов с минимальным безопасным набором инструментов для команд, файлов, git и тестов.
4. SQLite-хранилище для запусков и событий.
5. Управляющих FastAPI-эндпоинтах.
6. Базовых промптах и политике автономной работы через задачи.
