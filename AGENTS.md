# AGENTS.md

## Цель Проекта

Нужно разработать автономного агента для написания кода и выполнения смежных инженерных задач. Основной интерфейс взаимодействия с агентом — таск-трекер: требования передаются через задачи, уточнения обсуждаются в комментариях, прогресс отражается через статусы задач.

Агент должен быть реализован как Python-библиотека. Поверх библиотеки должен быть FastAPI-сервис, который предоставляет управляющие эндпоинты для инициации задач, просмотра истории попыток выполнения и статистики работы агента. Актуальное состояние задач читается из таск-трекера через MCP.

## Базовые Вводные

- В MVP не нужна авторизация пользователей и управление ролями.
- Доступ к LLM настраивается через URL, API-токен и имя модели.
- Таск-трекер подключается через MCP.
- Для локальной разработки нужен примитивный MCP-эмулятор таск-трекера.
- REST API нужен для управления и наблюдаемости, а не для диалогового общения.
- Нужен тестовый веб-интерфейс на React, запускаемый через Vite, для выполнения команд REST API.
- Frontend хранится в папке `frontend/`.
- Код агента хранится в `agent/simple_agent/`.
- MCP-эмулятор таск-трекера хранится в `emulator/task_tracker_emulator/`.
- Seed-данные и сценарии хранятся в `datasets/`.
- Тесты разнесены по доменам: `tests/agent/` и `tests/emulator/`.
- Python-зависимости в MVP управляются через `pip`.
- Все пользовательское взаимодействие по задаче идет через описание задачи, поля задачи и комментарии.
- Агент должен автономно выполнять задачи по коду и смежные инженерные задачи.
- Агент должен быть stateless по бизнес-логике: задачи, статусы, вопросы и заметки находятся в таск-трекере, изменения кода — в git/workspace, локальная БД хранит только наблюдаемость.
- После перезапуска агент должен выполнять reconciliation по live-состоянию таск-трекера, workspace и git, а не полагаться только на локальный `runs.status`.
- Правила workflow, git, коммуникации и definition of done должны задаваться конфигурацией проекта, а не зашиваться в runtime.
- Агент должен уметь получать дополнительные знания о проекте и задаче через project profile, task context и подключаемые материалы.
- Рабочие режимы агента должны быть конфигурируемыми. Runtime не должен хардкодить фиксированный список продуктовых режимов вроде "coding only" или "review only".
- Исполнительное ядро агента должно быть независимым от FastAPI.

## Возможности MVP

- Находить или принимать задачи.
- Захватывать задачу перед началом работы.
- Загружать контекст задачи из таск-трекера через MCP.
- Готовить изолированное рабочее пространство для задачи.
- Анализировать код, редактировать файлы, запускать команды, тесты, линтеры и форматтеры.
- Задавать вопросы через комментарии к задаче и переводить задачу в состояние ожидания.
- Публиковать прогресс и итоговый отчет через комментарии к задаче.
- Сохранять диагностические попытки выполнения, вызовы инструментов, события и базовую статистику.
- Собирать prompt из системных инвариантов, project profile, task context и дополнительных знаний.
- Позволять проекту настраивать режимы работы агента: реализация, ревью, документация, исследование, управление задачами и другие проектно-специфичные варианты.
- Предоставлять REST-эндпоинты для запуска работы, отмены работы, чтения observability-истории и статистики.
- Предоставлять тестовый React/Vite UI для ручной проверки REST API.
- Поддерживать локальный MCP-эмулятор таск-трекера для разработки и интеграционных тестов.
- Постепенно развить агента до уровня, где он может создавать git-коммиты в рабочей ветке.

## Предпочтительная Архитектура

- Python-пакет для исполнительного ядра агента.
- FastAPI-приложение как тонкий управляющий слой.
- MCP-клиент-адаптер для таск-трекера.
- Локальный реестр инструментов для файловой системы, shell, git, тестов и MCP-действий в таск-трекере.
- Менеджер рабочих пространств с отдельным рабочим пространством на задачу или checkout репозитория.
- Хранилище наблюдаемости на SQLite для MVP, с возможностью перехода на централизованный collector позже.
- Явный lifecycle выполнения попытки задачи и policy layer для claim/resume/final status.
- Слой project profile и prompt assembly для проектных правил, workflow, git-policy и дополнительных знаний.

## Инженерные Правила

- Исполнительное ядро должно оставаться пригодным для использования как библиотека без FastAPI.
- Интеграции нужно прятать за интерфейсами.
- Предпочитать небольшие заменяемые адаптеры вместо жесткой привязки к конкретным внешним системам.
- Не добавлять авторизацию, многопользовательские права, Kubernetes или распределенные рабочие процессы в MVP.
- Разработку функциональности вести в ветках; merge в `master` делать после подтверждения работоспособности.
- Не пушить в `master` из автономных или фоновых запусков агента.
- Установку Python-зависимостей и запуск тестов выполнять только в виртуальном окружении.
- Для установки Python-зависимостей использовать `pip`.
- Для активации виртуального окружения можно использовать алиас `vactivate`.
- Поддерживать локальный запуск без Docker, несмотря на целевое Docker-использование.
- Не скрывать ошибки от таск-трекера: неуспешный запуск должен оставлять диагностический комментарий.
- Не считать, что LLM-провайдер обязательно OpenAI; нужна адаптируемая клиентская абстракция.
- Комментарии к задаче являются частью продуктового поведения, а не отладочным журналом.
- Локальная SQLite-БД не является источником правды, lock-механизмом или рабочей памятью агента.

## Протокол Выполнения Этапа

Для каждого этапа обязательно использовать `planning/stage-checklist.md`.

Перед началом работы по этапу нужно:

- Проверить `planning/technical-debt.md` и убедиться, что нет блокеров для этапа.
- Проверить, что необходимые инструменты есть и работают или входят в scope этапа.
- Проверить `planning/feature-requests.md` и подсветить фиче-реквесты, которые могут повлиять на этап.
- Если фиче-реквест влияет на этап, сначала проработать его и согласовать решение.
- Кратко вывести план реализации этапа с декомпозицией, затрагиваемыми компонентами и проверками.
- Приступать к реализации только после подтверждения плана.

Во время этапа вся работа ведется в ветке. Нужно явно разделять два состояния:

- срез готов к проверке — реализация, тесты, smoke, документация и коммит выполнены в ветке, но merge в `master` еще не сделан;
- этап закрыт — после подтверждения пользователя ветка смержена в `master`, проверен `git status`, выполнен полный closeout.

По окончании среза нужно подвести предварительный итог и дождаться подтверждения. После merge в `master` нужно обязательно выполнить полный closeout: что удалось, что не получилось, какие проверки выполнены, появились ли новые пункты техдолга или фиче-реквесты, какие документы изменились, нужно ли ревью тестов, архитектуры или процесса разработки.

Этап завершен только когда рабочий срез проверен, документация актуальна, изменения смержены в `master` после подтверждения и closeout после merge передан пользователю. Нельзя отвечать только "смержено" или кратким статусом merge вместо closeout.

## Границы Первой Реализации

Первая реализация должна сфокусироваться на:

1. Исполнительном ядре агента, lifecycle выполнения и policy layer.
2. Интерфейсе MCP-клиента для таск-трекера.
3. Реестре инструментов с минимальным безопасным набором инструментов для команд, файлов, git, тестов и MCP-интеграций.
4. SQLite-хранилище наблюдаемости для попыток выполнения, событий и tool calls.
5. Управляющих FastAPI-эндпоинтах.
6. MCP-эмуляторе таск-трекера.
7. Тестовом React/Vite UI для REST API.
8. Сборке prompt из системных правил, project profile и task context.
9. Постепенном развитии агента до правок кода, проверок, diff-артефактов и git-коммитов с project-defined policy.
